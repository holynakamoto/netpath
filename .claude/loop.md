# netpath maintenance loop

You are running one iteration of the idle maintenance loop for netpath, a
diagnosis-first network incident investigator (Python, `uv`, pytest, ruff,
Textual TUI). Do the checks below in order, fix what is objectively broken,
and end the iteration. Each step has a machine-verifiable pass condition —
never judge "done" by reading code alone.

## Hard rules (read first)

- **Never create or push git tags.** A `v*` tag publishes to PyPI.
- **Never edit `dist/`, `uv.lock` (except via `uv sync`/`uv lock`), or
  `src/netpath/_version.py`** — the version file is generated by hatch-vcs.
- **Never force-push, rebase published history, or push to any branch you
  didn't create in this iteration** other than fast-forward pushes to `main`
  containing only your own validated fixes.
- If `git status` shows uncommitted changes you did not make this iteration,
  the user is mid-work: **do steps 1–3 read-only, report findings, and stop.**
- Cap each iteration at roughly one focused fix. If a diagnosis balloons
  (multi-file refactor, unclear root cause after two attempts), stop, leave
  the tree clean (`git stash` your partial work with a descriptive message or
  discard it), and write the findings to `.claude/loop-notes.md` so the next
  iteration or the user can pick it up.

## Iteration steps

### 1. Sync state

```
git fetch origin
git status
gh run list --branch main --limit 5
gh pr list --state open
```

If local `main` is behind `origin/main` and the tree is clean, fast-forward it.

### 2. CI triage (highest priority)

If the most recent CI run on `main` failed:

1. `gh run view <id> --log-failed` to get the failing step and error.
2. Reproduce locally with `make validate` (runs `uv sync --extra dev`,
   `pytest -q`, `ruff check .`). CI also builds sdist/wheel and rejects
   artifacts containing `.venv/`, `__pycache__/`, or `.pyc` files — if that
   step failed, reproduce with `uv run python -m build --sdist --wheel` and
   fix the packaging config, then delete your local `dist/` additions.
3. Apply the **minimal** fix. Remember CI tests Python 3.9–3.13, so avoid
   3.10+ syntax (`match`, `X | Y` unions in runtime positions, etc.).
4. Verify: `make validate` exits 0.
5. Commit with a message describing the failure and fix, push to `main`,
   then confirm the new CI run goes green before treating this as resolved
   (`gh run watch` or re-check on the next iteration).

### 3. Open PRs

For each open PR with unresolved review comments or failing checks:
address the comments with minimal commits to the PR branch, reply to and
resolve the threads, and re-run `make validate` before pushing. Do not merge
PRs — leave merging to the user.

### 4. Local quality gate

Even if CI is green (it only sees pushed commits):

```
make validate
```

- **pytest failures** → fix the code or the test, whichever is actually
  wrong; re-run until `pytest -q` exits 0.
- **ruff findings** → fix them; `ruff check .` must exit 0. Prefer real fixes
  over `# noqa`; only suppress with a comment explaining why.

If you made fixes here, commit and push them (subject to the hard rules).

### 5. Done check

The iteration is complete when all of these hold:

- Latest CI run on `main` is green (or a fix for it is pushed and pending).
- `make validate` exits 0 locally.
- No open PR has unresolved review comments you can address.
- Working tree is clean.

When everything already passes at step 5, say so in one short summary line
and end the iteration immediately — do not invent work, refactor
speculatively, bump dependencies, or "improve" passing code. An empty
iteration is a successful iteration.

## Reporting

End every iteration with a 2–4 line summary: what was checked, what was
fixed (with commit hashes if you pushed), and anything deferred to
`.claude/loop-notes.md`.
