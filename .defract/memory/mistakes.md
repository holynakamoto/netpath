# Mistake Patterns

## Mistakes

- [01KWF7TABHDVA5M83VCSWG95WN] **- **Stale TypedDict fields after removing keys from `_measure()`** — After ...** -- - **Stale TypedDict fields after removing keys from `_measure()`** — After Phase 2 replaced `_trace_error`, `_iperf_error`, `_speedtest_error` with the unified `probe_errors` dict, the `MeasurementResult` TypedDict in `types.py` still declared those three as optional fields (lines 43, 49, 52). A maintainer reading the TypedDict would infer those keys can still appear in results when they cannot. **Why:** TypedDict fields and the actual dict are two separate things; removing a key from the producing function doesn't automatically remove it from the type annotation. **How to apply:** When removing keys from `_measure()` (or any function with a matching TypedDict), update the TypedDict in `types.py` in the same commit/phase. Treat the TypedDict as the canonical schema — if it diverges from the implementation, the type annotations become misleading rather than protective. - [01KWDAP76RH5KQSR4Z3E9TPKGE] **- **Test isolation failure: calibrated-loss unit test also passes via rate-li...** -- - **Test isolation failure: calibrated-loss unit test also passes via rate-limit suppression** — `test_calibrated_loss_few_probes_no_alarm` (tests/test_diagnosis.py:91) passes for two independent reasons: (1) calibration raises the threshold to 5% so 3% doesn't alarm, AND (2) the clean downstream hop triggers rate-limit suppression. The test does not isolate calibration alone. **Why:** When two separate features both prevent an alarm, a single test can't distinguish which one is doing the work — if calibration regresses, the test still passes via suppression. **How to apply:** For any test that verifies a threshold or calibration feature, ensure the scenario rules out alternative passing mechanisms. Companion test: downstream hops also carry 3% loss (so rate-limit suppression can't apply) — Healthy verdict then requires calibration alone. - [01KWHBTPB6QM812RG7N28MDY8J] **- **Worktree sync conflict from `** -- - **Worktree sync conflict from `.venv` `__pycache__` files after running tests** — Running `pytest` inside a worktree generates `.pyc` files under `.venv/lib/.../site-packages/rich/__pycache__/`. When defract later attempts to sync the worktree from `main`, git reports "Your local changes to the following files would be overwritten by merge" for those `.pyc` files. This caused 3 consecutive sync failures (`worktree_sync_conflict`) requiring a recovery agent to resolve. **Why:** The `.venv` directory sits inside the worktree root and its compiled Python cache files are not in `.gitignore` (or the ignore rules don't cover site-packages pycache). **How to apply:** Before running any test or install command inside a worktree, confirm `.venv` or at minimum `*.pyc`/`__pycache__` under `.venv` is covered by `.gitignore`. If sync conflicts appear on `.pyc` paths, `git checkout -- <path>` those files before retrying the merge. [source: task-readme-atlas-key-docs-asn-names-in-01kwh0zv63hp, importance: 0.65].
